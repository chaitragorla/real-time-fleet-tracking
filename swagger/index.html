NestJS + MongoDB Backend

AddWise GPS API

Compatibility documentation for the migrated backend. Legacy `/v1` routes are preserved while MongoDB, Mongoose indexes, JWT tokens, and DTO validation power the new service layer.

Authentication

POST/v1/auth/send-otp

Body: { "phoneNumber": "9876543210", "name": "Optional", "role": "customer" }

200: { success, message, phoneNumber, userExists, smsStatus, smsDetails }

400: invalid or missing phone number.

POST/v1/auth/verify-otp

Body: { "phoneNumber": "9876543210", "otp": "123456", "name": "Optional", "role": "customer", "email": "optional@example.com" }

200: { success, message, isNewUser, userType, user, sessionToken, accessToken, otpVerified }

401: invalid or expired OTP.

GET/v1/auth/profile/:sessionToken

Returns the legacy super-admin profile for a `session_...` token.

PUT/v1/auth/profile/:sessionToken

Body: { "name": "New Name" }. Updates a legacy profile.

POST/v1/auth/cleanup-otps

Deletes expired OTP records. OTPs also have a MongoDB TTL index.

GPS Tracking

POST/v1/gps-signal/update-location

Body: { device_code, device_m2m_number, latitude, longitude, accuracy? }

Validates that the 13-digit M2M number belongs to the device, stores a GeoJSON point, and returns the legacy insert shape.

GET/v1/gps-signal/current-location

Returns the in-memory current location compatibility payload.

GET/v1/gps-signal/:device_code

Stores the current location for an active device and returns `{ device_code, latitude, longitude, timestamp, user_id, device_name, status }`.

GET/v1/gps-signal/:device_code/history

Returns chronological route history: `{ status, device_code, count, data }`.

GET/v1/gps-signal/device/:deviceCode/data

Frontend map endpoint returning `{ success, data, count }`.

DELETE/v1/gps-signal/device/:deviceCode/clear

Deletes all GPS rows for a device and returns `deletedCount`.

POST/v1/gps-signal/device/:device_code/active

Body: { "is_active": true }. Enables or disables tracking.

Devices, SMS, System

GET/v1/devices/active

Returns active allocated devices for monitoring and simulations.

GET/v1/devices

Returns { success, count, data } for all Mongo-backed devices.

POST/v1/devices

Body: { "count": 1 } or { "devices": [{ "device_code": "ABC", "qr_code": "ABC" }] }.

GET/v1/devices/code/:deviceCode

Fetches one device by code.

GET/v1/devices/owner/:ownerId

Returns devices allocated to a customer.

GET/v1/devices/received/:userId

Returns device shares received by a customer, including owner and device records.

GET/v1/devices/sent/:ownerId

Returns device shares sent by an owner.

PATCH/v1/devices/:deviceCode

Body: { device_name?, device_icon?, device_m2m_number?, is_active? }.

PATCH/v1/devices/id/:legacyId

Updates a device by stable numeric legacy ID.

POST/v1/devices/:deviceCode/allocate

Body: { allocated_to_customer_id, allocated_to_customer_name, device_name?, device_icon? }.

POST/v1/devices/:deviceCode/share

Body: { "phone_number": "9876543213" }. Creates or reuses a Mongo device share.

DELETE/v1/devices/shares/:shareId

Revokes shared-device access.

POST/v1/devices/id/:legacyId/unassign

Unassigns one device and removes its share records.

POST/v1/devices/owner/:ownerId/unassign-all

Unassigns all devices owned by a customer and removes related share records.

DELETE/v1/devices/id/:legacyId

Deletes a device by legacy ID.

DELETE/v1/devices/:deviceCode

Deletes a device by code.

GET/v1/users?role=customer

Lists users by role: customer, admin, or superadmin.

GET/v1/users/by-phone/:phoneNumber?role=customer

Finds a Mongo user by phone and optional role.

POST/v1/users

Body: { phone_number, full_name?, email?, role?, employee_id? }.

GET/v1/users/login-logs

Returns employee login audit records.

POST/v1/users/login-logs

Body: { "employee_id": "EMP001" }. Creates an admin login audit record.

DELETE/v1/users/:legacyId?role=customer

Deletes a user by stable numeric legacy ID and optional role.

POST/v1/sms/send

Body: { "phoneNumber": "9876543210", "message": "Your OTP is 123456" }

GET/health

Returns service status, MongoDB connection state, and SMS/OTP availability.
